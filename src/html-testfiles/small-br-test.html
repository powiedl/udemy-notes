describe.only und test.only hat eine sehr lange Überschrift - wird sie umgebrochen und wenn ja wo?

describe.only oder test.only schreibt (dann wird nur der .only Teil ausgeführt). Das kann manchmal hilfreich sein, weil es zu "Interferenzen" zwischen einzelnen Tests kommen kann. Auf diese Weise kann man recht schnell feststellen, ob der Test, der nicht wie erwartet funktioniert, falsch ist oder ob es vielleicht zu so einer Interferenz kommt.

Debugger für Test aktivieren

Dazu muss man in der package.json im scripts Block ein weiteres Kommando einfügen:

"test:debug": "react-scripts --inspect-brk test --runInBand --no-cache"

Danach kann man - wo man den debugger aktivieren will - ein debugger Statement einfügen. Das kann in einem Test sein - aber auch im Sourcecode der getesteten Komponente/Funktion. Dann den Test mit npm run test:debug starten. Im Browser dann about:inspect aufrufen, und man kann an der Stelle debuggen.

84. Order of Execution

Jest - Ausführungsreihenfolge von Testfiles

Jest führt zuerst alles in einem Testfile (von oben nach unten) aus - ohne die eigentlichen Tests aufzurufen. Erst danach werden die einzelnen Tests (ebenfalls in der Reihenfolge von oben nach unten im Testfile) ausgeführt. Das kann manchmal störend sein. Die Lösung ist, dass man Tests (und auch weiteren Code) gruppieren kann. Damit werden die Gruppen jeweils sauber ausgeführt.

Das macht man mit der Funktion describe. Diese funktioniert genauso wie test, aber enthält in der Funktion (2. Parameter) im Normalfall mehrere test() Aufrufe (und ev. auch Initialisierungen für diese "Gruppe"):

  1. describe('when user is not <button>Click me</button> signed in',() => {
  2.   createServer(...) // fuer Response, wenn der User nicht angemeldet ist
  3.   test('sign in button is visible',() => { ... }
  4.   test('sign out button is not visible',() => { ... }
  5. });
  6.  
  7. describe('when user is signed in',() => {
  8.   ...
  9. });

Ein beforeAll, afterAll ..., dass in einem describe steht, gilt auch nur für diesen.

79. An Issue with Fake Handlers

Common Pattern - die handler in einem File zu zentralisieren

Oft findet man die Empfehlung alle handler in einem zentralen File zusammenzufassen. Auf den ersten Blick ist das eine gute Idee. Der wesentliche Nachteil dieser Methode ist, dass alle Tests mit den gleichen Responses für ein- und denselben Request klarkommen müssen. Das ist oft nicht gewünscht - und in so einem Fall ist es dann keine gute Idee, die handler zu zentralisieren - sondern in so einem Fall ist es "besser" sie in den jeweiligen Test-Files immer wieder neu zu schreiben.
