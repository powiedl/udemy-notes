AuthJS - authorized callback

Man kann einen authorized callback in der NextAuth(config) aufnehmen. Dieser wird aufgerufen, wenn ein User eine Authorization benötigt. Dabei muss das als NextJS-middleware aufgesetzt werden, d. h. im middleware.ts Skript:

Im @/auth.ts muss man dann den authorized callback definieren (in config.callbacks):

  1. authorized({request,auth}:any) {
  2.   const { authToken } = (await request.json()) ?? {}
  3.   // If the request has a valid auth token, it is authorized
  4.   const valid = await validateAuthToken(authToken)
  5.   if(valid) return true
  6.   return NextResponse.json("Invalid auth token", { status: 401 })
  7. }

Und es muss entweder true (wenn die Response nicht geändert werden soll) oder ein NextResponse zurückliefern. Bei true, wird der Request zur nächsten Middleware durchgelassen, bei NextResponse wird dieser zurückgegeben (siehe Beispiel in auth.ts authorized callbacks).

Auch in NextJS 15 werden route params als Verzeichnisse abgebildet, damit ist der Name des Route Parameters in [] zu schreiben, d. h. der Ordner [id] definiert einen Route Parameter id. Um diesen in der Page auswerten zu können, geht man so vor:

  1. const ProductPage = async({params}:{params:Promise<{id:string}>;}) => {
  2.   const {id} = await params;
  3.   console.log(id);
  4. }


Wichtig: Params sind mit NextJS 15 asynchron, daher muss man sie awaiten - und damit muss die Komponente auch async sein. Und params sind eben ein Promise (im obigen Beispiel gibt es nur einen Route Parameter id, der vom Typ string [wie vermutlich jeder Route Parameter?] ist).


Die Query Parameter werden "genauso" übertragen, allerdings im Objekt searchParams, d. h. ein Queryparameter q funktioniert so:

  1. const ProductPage = async({searchParams}:{searchParams:Promise<{q:string}>}) => {
  2.   const {q} = await searchParams;
  3.   ...
  4. }

  4.   name:z.string().min(3),
  5.   bidPrice:currency,
  6.   askPrice:currency,
  7. });

Prisma Schema - Fortsetzung

Bei price gibt man an, dass die Zahl 12 Stellen vor dem Komma haben darf und 2 danach. Dafür verwendet prisma die Library Decimal.js (ACHTUNG: Das wird intern dann als String "gespeichert")

Wenn man das Schema ändert muss man danach den @prisma/client neu generieren. Das geht mit npx prisma generate. Weil wir das auch am Server machen müssen, muss man in package.json in scripts ein Script "postinstall": "prisma generate" aufzunehmen.


Um dann auch die Datendank selbst anzupassen muss man migrate ausführen. Man kann dem Migrate auch einen Namen (im folgenden Beispiel) init mitgeben:

npx prisma migrate dev --name init

Das legt im prisma-Verzeichnis ein Verzeichnis migrations an, wo dann die SQL Statements gespeichert werden (die gegen die Datenbank ausgeführt werden).

20. Prisma Models & Migrations

Prisma Schema

Das Schema bezeichnet alle Modelle (und ihre Beziehung untereinander gemeinsam). Ein Modell ist dann eine Tabelle in der Datenbank.

Beispiel für ein Produkt-Modell (Auszug):

  1. model Product {
  2.   id        String   @db @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  3.   slug      String   @unique(map: "product_slug_idx")
  4.   banner    String?
  5.   images    String[]
  6.   price     Decimal  @default(0) @db.Decimal(12,2)
  7.   createdAt DateTime @default(now()) @db.Timestamp(6)
  8. }

Das id Attribut ist "typisch" - es definiert einen String (der im Detail vom Typ Uuid ist) und es generiert automatisch eine zufällige UUID in der Datenbank.

Bei slug stellt man sicher, dass es über alle Produkte hinweg eindeutig sein muss (mit @unique - darum kümmert sich dann die Datenbank). Und mit map: ... legt man auch den Index fest, der sich darum kümmert.

Bei banner handelt es sich um ein optionales Feld, d. h. es muss nicht vorhanden sein.

Bei images handelt es sich um ein Array von Strings.

12. Theme Mode Toggle

NextJS - Theme

Wenn man in einer NextJS Anwendung Theme Unterstützung aktivieren will sind folgende Schritte notwendig:

npm i next-themes

An zentraler Stelle (z. b. im Main Layout):

  1. import {ThemeProvider } from 'next-themes';
  2. ...
  3.   <html suppressHydrationWarning>
  4.   ...
  5.     <ThemeProvider attribute='class' defaultTheme='light' enableSystem disableTransitionOnChange>
  6.       {children}
  7.     </ThemeProvider>

suppressHydrationWarning muss man setzen, weil next-themes window verwendet (was am Server nicht verfügbar ist).

attribute='class' legt fest, dass das Theme über eine Klasse (.dark bzw. .light) gesteuert wird. enableSystem erlaubt, dass man das System Theme verwendet. Und disableTransitionOnChange verhindert, dass Transistioneffekte angewendet werden, wenn man zwischen den Themes wechselt.

ACHTUNG: defaultTheme gilt nur für NEUE User, sobald jemand auf die Seite geht, wird das Theme in localStorage (theme) gespeichert!
